# Data Model Changes: Миграция из Bitrix в Payload CMS

## Введение
Для обеспечения идемпотентности скрипта миграции и предотвращения дубликатов при повторных запусках, необходимо добавить внешние идентификаторы Bitrix в существующие коллекции Payload.

## Изменения в коллекциях

### 1. Категории (`categories`)
- `bitrixId`:
  - **Type**: `number`
  - **Label**: "ID в Bitrix"
  - **Required**: `false` (только для мигрированных)
  - **Unique**: `true`
  - **Index**: `true`
  - **Admin**: `position: 'sidebar'`

### 2. Товары (`products`)
- `bitrixId`:
  - **Type**: `number`
  - **Label**: "ID в Bitrix"
  - **Required**: `false`
  - **Unique**: `true`
  - **Index**: `true`
  - **Admin**: `position: 'sidebar'`

### 3. Медиа (`media`)
- `bitrixId`:
  - **Type**: `number`
  - **Label**: "ID в Bitrix"
  - **Required**: `false`
  - **Index**: `true`
  - **Admin**: `description: 'Используется для дедупликации загрузок из Bitrix'`

## Логика сопоставления (Relationships)
- **Родительские категории**: При миграции категории с `IBLOCK_SECTION_ID`, скрипт должен найти ID существующей записи Payload, у которой `bitrixId` соответствует значению родителя в Bitrix.
- **Товары и категории**: После создания товаров, скрипт связывает их с коллекцией `categories` по аналогичному принципу сопоставления `bitrixId`.
- **Товары и изображения**: Каждое изображение из Bitrix проверяется на наличие в коллекции `media` по `bitrixId`. Если отсутствует — скачивается и создается новая запись `media`, которая затем добавляется в массив `gallery` товара.

## Валидация
- **Слизни (Slugs)**: При создании новых записей из Bitrix необходимо генерировать уникальные `slug` на основе названия (title) с использованием встроенных механизмов Payload или библиотеки `slugify`.
