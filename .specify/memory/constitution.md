# Accurate Payload Constitution

## Core Principles

### I. Payload-First Architecture
Использование Payload CMS 3.0 как основного ядра для управления данными и контентом. Все сущности (продукты, заказы, страницы) должны определяться через коллекции и глобалы Payload. Кастомизация интерфейса и логики должна следовать паттернам Payload.

### II. Next.js 15 & React 19 Standards
Применение современных возможностей Next.js App Router и React Server Components. Архитектура должна обеспечивать высокую производительность, SEO-оптимизацию и четкое разделение ответственности между серверной частью CMS и клиентским фронтендом.

### III. Обязательная типизация и стандарты кода
Строгое использование TypeScript. После любых изменений схемы Payload необходимо запускать генерацию типов (`pnpm generate:types`). Код должен соответствовать установленным правилам ESLint и Prettier.

### IV. Русскоязычная коммуникация
Весь процесс разработки, включая комментарии в коде, сообщения коммитов (по возможности), общение с агентом и документацию, должен вестись на русском языке. Это обеспечивает единообразие и понимание контекста всеми участниками.

### V. Тестирование и верификация (NON-NEGOTIABLE)
Каждая новая функциональность должна сопровождаться тестами (интеграционными или E2E). Перед деплоем обязательна проверка типов (`tsc`), линтинг и успешная сборка проекта.

## Стек технологий и инфраструктура

- **База данных**: PostgreSQL (Supabase) через адаптер `@payloadcms/db-postgres`.
- **Медиа**: Vercel Blob Storage.
- **Платежи**: Stripe API.
- **Стилизация**: Tailwind CSS и Shadcn UI.
- **Среда**: Node.js >= 20.

## Процесс развертывания (Deployment Workflow)

Деплой осуществляется через Vercel CLI. Критически важным является удаление 'dev' записей из таблицы миграций перед каждым деплоем (`DELETE FROM payload_migrations WHERE name = 'dev'`), чтобы избежать зависания процесса сборки на Vercel.

## Governance

Конституция имеет приоритет над всеми остальными практиками разработки. Любые изменения в архитектуре или ключевых принципах должны фиксироваться здесь с обновлением версии. Все PR должны проверяться на соответствие этим принципам.

Все решения по архитектуре должны учитывать масштабируемость и безопасность eCommerce-решений. Использование сторонних библиотек требует обоснования и проверки на совместимость с Payload 3.0.
